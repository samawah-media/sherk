## 3. Assumptions

| الافتراض | سبب الافتراض | أثره إذا كان خاطئا |
| --- | --- | --- |
| التوسع التجاري خارج سماوة يأتي بعد إثبات التشغيل الداخلي | شريك SaaS خارجي لكن الإطلاق الأول داخل سماوة | قد تتغير أولويات go-to-market إذا طلب المالك إطلاقا عاما مبكرا |
| أول تجربة ستكون مع عميلين حاليين | الوثائق القديمة تذكر عميلين | قد تتغير مقاييس النجاح |
| مستوى التفصيل المطلوب "تفصيلي عملي" | خطة معتمدة افتراضيا | قد يحتاج توسعة أو اختصار |
| مدير الحساب دور وسط بين الفريق والإدارة | ظاهر من واقع المنتج | قد تتغير مصفوفة الصلاحيات |
| الباقة تعرض كميات أكثر من فوترة مالية متقدمة | advanced billing خارج V1 | قد تحتاج شاشة مالية أكبر |
| الإشعارات في V1 داخل المنصة وربما بريد لاحقا | تجنب التكاملات المبكرة | قد يقل تبني العميل إذا اعتمد على واتساب |
| العميل يريد أقل عدد شاشات ممكن | AGENTS.md يؤكد بساطة العميل | إذا احتاج تفاصيل أكثر قد تتوسع البوابة |
| الفريق يرى على الأقل مخرجاته المسندة | صلاحيات الفريق محدودة | إذا احتاج رؤية أوسع تتغير سياسات الوصول |
| versioning بسيط يكفي في MVP | AGENTS.md يطلب versioning بسيط على الأقل | إذا كانت الملفات كثيرة قد نحتاج نموذج نسخ أقوى |
| at-risk thresholds تحدد لاحقا حسب نوع المخرج | غير مثبتة الآن | لا يمكن بناء حساب SLA تفصيلي قبل القرار |

## 4. High-Priority Open Questions

### أسئلة حسمتها Phase 02 Operating Model

| السؤال السابق | القرار الجديد | التصنيف |
| --- | --- | --- |
| هل شريك SaaS خارجي أم منصة داخلية لسماوة أولا؟ | شريك SaaS خارجي مع إطلاق داخلي أولا لدى سماوة. | Confirmed |
| ما نموذج tenancy في V1؟ | Tenant يمثل الوكالة أو الجهة المشغلة، وسماوة أول Tenant، وكل Tenant يدير عدة عملاء مع عزل كامل بينهم. | Confirmed |
| هل كل المخرجات تحتاج تعميد داخلي؟ | التعميد الداخلي إلزامي للمخرجات الموجهة للعميل. | Confirmed |
| هل كل المخرجات تحتاج اعتماد عميل؟ | اعتماد العميل قابل للضبط حسب قالب أو نوع المخرج. | Confirmed |
| متى يحتسب المخرج ضمن الباقة؟ | يحجز عند الإنشاء، يستهلك عند التسليم النهائي، ويعاد الحجز عند الإلغاء. | Confirmed |
| هل مدير الحساب يستطيع اعتماد داخلي؟ | مدير الحساب ينسق ويوزع ويتابع، ولا يعتمد نهائيا إلا بصلاحية صريحة. | Confirmed |

## 7. Demo-Derived Observations

### ما يمكن اعتماده كإلهام

- واجهات منفصلة للعميل والفريق والإدارة.
- Sidebar لكل دور.
- Kanban داخلي.
- Drawer للمخرج.
- Tabs للملفات والتعليقات والجودة والسجل.
- Badges للحالة وSLA.
- RTL عربي وواجهة SaaS.

### ما لا يعتمد من الديمو

- أي زر أو إجراء يسمح بتجاوز التعميد الداخلي كمسار عادي.
- أي حالة SLA تبقي طلب تعديل العميل على مسؤولية العميل بعد عودة العمل للفريق.
- أي عرض ملفات للعميل دون visibility وtenant.
- أي بيانات وهمية مثل أسماء العملاء أو الأرقام كمصدر متطلبات.

| التصنيف | المعلومة |
| --- | --- |
| Confirmed | الديمو مرجع UI/UX فقط وليس مصدر متطلبات نهائي. |
| Assumed | نمط Drawer مناسب لمرحلة MVP. |
| Open Question | هل يرغب المالك بالحفاظ على نفس أسلوب الألوان والدروارات في التصميم النهائي؟ |

## 8. Methodology Notes

استخدمت هذه المرحلة:

- Product Manager Skills كإطار لصياغة PRD وPersonas وJTBD وUser Stories وRoadmap.
- BMAD كمراجعة متعددة المنظور دون إنشاء وكلاء أو ملفات تنفيذية.
- Lean Canvas لاختبار الافتراضات التجارية المبكرة.
- Shape Up لتقسيم التنفيذ لاحقا إلى Cycles.
- Spec Kit وAgent OS كمرحلة لاحقة للـ Specs والمعايير.
- C4 وarc42 وADR وOWASP ASVS وAWS/Azure Multitenancy كمداخل للمرحلة المعمارية والأمان اللاحقة.

ملاحظة Phase 02: بناء على قرار المالك في 2026-06-22، تستخدم مرحلة Operating Model & Core Business Rules إطار Product Manager Skills وBMAD فقط، ولا تنشئ Specs أو ADRs أو مخططات معمارية تقنية.
