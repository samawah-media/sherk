## 1. الغرض

هذه الوثيقة مسودة Product-level للصلاحيات. لا تعد RLS policy ولا تصميم قاعدة بيانات. الهدف هو تثبيت من يرى ماذا ومن يستطيع فعل ماذا قبل مرحلة المعمارية وSpec Kit.

| التصنيف | المعلومة |
| --- | --- |
| Confirmed | Tenant يمثل الوكالة أو الجهة المشغلة، وسماوة هي أول Tenant. |
| Confirmed | كل Tenant يستطيع إدارة عدة عملاء مع عزل كامل بينهم. |
| Confirmed | العميل لا يرى تعليقات داخلية أو بيانات عميل آخر. |
| Assumed | الأدوار يمكن أن تطبق كمجموعة roles/permissions مرنة لاحقا. |
| Open Question | هل يوجد دور Super Admin للمنتج خارج سماوة في V1؟ |

## 2. مبادئ الصلاحيات

- أقل صلاحية ممكنة بشكل افتراضي.
- العميل يرى بيانات عميله فقط.
- الفريق يرى المخرجات المصرح له بها.
- الإدارة ترى العملاء المصرح بهم.
- كل إجراء اعتماد أو تغيير حالة أو تنزيل ملف حساس يجب أن يتحقق من الدور والنطاق.
- الصلاحية لا تغني عن tenant isolation.
- Audit Log إلزامي للقرارات الحساسة.

## 3. الأدوار الأولية

### 3.1 أدوار الإدارة

- `samawah_admin`: صلاحية إدارية عليا ضمن Tenant سماوة كأول مشغل.
- `project_manager`: يدير العملاء، المخرجات، التعميد، الإرسال، الإغلاق.
- `marketing_manager`: يراجع الجودة ويعمد داخليا حسب الصلاحية.
- `account_manager`: يدير علاقة العميل ويتابع المخرجات وقد يرسل للعميل بعد التعميد.

### 3.2 أدوار فريق العمل

- `content_writer`: ينفذ مخرجات المحتوى.
- `designer`: ينفذ التصميم والملفات البصرية.
- `motion_designer`: ينفذ الفيديو/الموشن إذا اعتمد كدور.
- `performance_specialist`: يعمل على التقارير والتحليل.
- `publisher`: ينفذ النشر اليدوي خارج المنصة إذا وجد كعملية لاحقة.

### 3.3 أدوار العميل

- `client_admin`: يدير مستخدمي جهة العميل وجهات الاعتماد.
- `client_approver`: يعتمد أو يطلب تعديل ضمن الصلاحية.
- `client_viewer`: يشاهد المخرجات والملفات المصرح بها دون اعتماد.

## 5. قواعد رؤية العميل

العميل يرى:

- مخرجات عميله فقط.
- المخرجات المرسلة له بعد التعميد الداخلي.
- الملفات ذات visibility مناسبة مثل `client_visible`, `client_uploaded`, `final_delivery`, `contract_file`.
- تعليقات العميل والتعليقات العامة المسموح بها.
- سجل القرارات الخارجية المناسب.

العميل لا يرى:

- `internal_comment`.
- Checklist الجودة.
- النسخ الداخلية غير المرسلة.
- ملفات `internal_only`.
- عملاء آخرين.
- Kanban الداخلي.
- أسماء أو تفاصيل داخلية إذا قررت الإدارة إخفاءها.

## 6. قواعد رؤية الفريق

الفريق يرى:

- مخرجات مسندة له أو مصرح له بها.
- الملفات اللازمة للعمل.
- التعليقات الداخلية.
- ملاحظات العميل المرتبطة بالمخرج إذا كان مصرحا.
- SLA والحالة.

الفريق لا يرى افتراضيا:

- ملفات أو مخرجات عملاء خارج نطاقه.
- إعدادات العقود الحساسة.
- صلاحيات الإدارة.
- قرارات لا تخص عمله إذا تم تقييدها.

| التصنيف | المعلومة |
| --- | --- |
| Confirmed | لا Query أو API دون tenant/client scope. |
| Assumed | يمكن تقليل رؤية الفريق حسب assignment لتقوية الخصوصية. |
| Open Question | هل يريد المالك تشغيل الفريق برؤية كل عملاء سماوة أم رؤية حسب العميل/المهمة؟ |

## 7. قواعد رؤية الإدارة

الإدارة ترى:

- العملاء المصرح بهم.
- مخرجاتهم.
- SLA والتأخيرات.
- التعميدات والاعتمادات.
- الملفات حسب الصلاحية.
- Audit Log.

الإدارة تستطيع:

- التوزيع والمراجعة والتعميد والإرسال والإغلاق.
- تعديل SLA فقط بسبب موثق وصلاحية واضحة.
- إعادة فتح مخرج عند الحاجة مع سبب.

## 8. سياسات حساسة يجب تثبيتها لاحقا

- سياسة tenant/client isolation.
- سياسة file access.
- سياسة comments visibility.
- سياسة approval links إن وجدت.
- سياسة Audit Log retention.
- سياسة تعديلات SLA اليدوية.

هذه السياسات تنتمي إلى مرحلة المعمارية والأمان، لكنها مذكورة هنا كمتطلبات منتج.

## 9. أسئلة مفتوحة

- هل يوجد دور Super Admin للمنتج خارج سماوة في V1؟
- هل يوجد مستخدم من نفس جهة العميل يملك حق إدارة باقي المستخدمين؟
- هل يسمح للعميل برفع ملفات في V1؟
- هل يستطيع client_viewer كتابة تعليق غير رسمي؟
- هل يرى الفريق كل التعليقات العميلية أم فقط للمخرجات المسندة؟
- هل يحتاج Audit Log لواجهة تصدير في V1؟
