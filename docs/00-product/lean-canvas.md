## 1. الغرض

هذا الملف يلخص نموذج المنتج في صفحة عملية واحدة، لا كخطة مالية نهائية. الهدف هو كشف الافتراضات المبكرة التي يجب اختبارها قبل الدخول في Specs أو معمارية أو تنفيذ.

| التصنيف | المعلومة |
| --- | --- |
| Confirmed | نستخدم Lean Canvas بصورة مختصرة ضمن مرحلة التخطيط الأولى. |
| Assumed | بعض عناصر القنوات والإيرادات والتكاليف افتراضية لأنها لم تثبت ببيانات سوقية بعد. |

## 2. Customer Segments

### الشرائح الأساسية

- وكالات التسويق التي تدير مخرجات مستمرة مع أكثر من عميل.
- فرق إدارة المحتوى والخدمات داخل الشركات التي تحتاج اعتماد عميل أو إدارة داخلية.
- عملاء الوكالات الذين يحتاجون متابعة مبسطة دون الدخول في أدوات تشغيل معقدة.

### Early Adopters

- سماوة كأول مشغل وبيئة تجربة.
- عميلان حاليان أو أكثر لديهم عقود/باقات ومخرجات مستمرة.
- فريق داخلي صغير نسبيا يحتاج Kanban ومراجعة واعتماد.

| التصنيف | المعلومة |
| --- | --- |
| Confirmed | الأطراف الثلاثة هي العميل، فريق العمل، الإدارة. |
| Assumed | أول استخدام واقعي سيكون داخل سماوة ومع عملاء حاليين. |
| Open Question | من يدفع فعليا مستقبلا: الوكالة، العميل النهائي، أم نموذج أبيض White-label؟ |

## 3. Problems

### المشكلة 1: تشتت سياق المخرج

المخرج الواحد يعيش غالبا بين محادثات، ملفات، جداول، وتعليقات متفرقة. هذا يجعل معرفة النسخة الصحيحة أو القرار الأخير عملية بطيئة ومعرضة للخطأ.

### المشكلة 2: ضعف التوثيق والاعتماد

الاعتمادات وطلبات التعديل قد تتم في واتساب أو بريد أو مكالمة. عند الخلاف، لا يوجد دائما سجل واضح: من وافق؟ متى؟ على أي نسخة؟

### المشكلة 3: غموض مسؤولية التأخير

عندما ينتظر الفريق رد العميل، قد يظهر العمل متأخرا وكأنه تأخير داخلي. هذا يضر الثقة ويصعب قياس SLA بعدل.

### المشكلة 4: خطر تسرب الداخلي للعميل

بدون فصل صارم بين الداخلي والخارجي، قد يرى العميل تعليقات جودة، نسخا غير جاهزة، أو ملاحظات لا يفترض ظهورها.

| التصنيف | المعلومة |
| --- | --- |
| Confirmed | SLA يجب أن يتوقف عند انتظار العميل. |
| Confirmed | لا يجب كشف التعليقات الداخلية للعميل. |
| Assumed | حجم الألم كاف لتبرير منتج SaaS إذا تكرر لدى وكالات مشابهة. |
| Open Question | ما أكثر ألم حالي لدى سماوة: الاعتماد، الملفات، SLA، أم متابعة المتبقي من الباقة؟ |

## 4. Existing Alternatives

- واتساب: سريع لكنه غير منظم، ضعيف البحث والتوثيق، وسهل خلط الداخلي بالخارجي.
- Google Drive/Dropbox: جيد للملفات، لكنه لا يدير الحالات والتعميد وSLA.
- Excel/Google Sheets: مناسب للمتابعة اليدوية، لكنه هش في الصلاحيات والنسخ وسجل النشاط.
- Trello/ClickUp/Asana: جيد للتشغيل الداخلي، لكنه ليس موجها لبوابة عميل مبسطة وعزل تعليقات داخلية.
- BrightBean وأدوات مشابهة: مرجع UX ووظيفي، وليس قاعدة بناء مباشرة في V1.

| التصنيف | المعلومة |
| --- | --- |
| Confirmed | BrightBean/Postiz/Planka مراجع فقط وليست قواعد بناء مباشرة. |
| Assumed | البدائل الحالية لدى العملاء والوكالات مشابهة لما سبق. |
| Open Question | هل لدى سماوة أداة حالية يجب مراعاتها في ترحيل البيانات لاحقا؟ |

## 5. Unique Value Proposition

> شريك يحول المخرجات المتفق عليها من محادثات وملفات متفرقة إلى دورة تسليم موثقة تربط الفريق والإدارة والعميل، مع اعتماد داخلي، موافقة عميل، SLA عادل، وملفات وتعليقات مفصولة حسب الصلاحية.

### High-level Concept

> "Trello + بوابة عميل + اعتماد/SLA" لكن مصمم خصيصا للمخرجات المتفق عليها، وليس للطلبات العشوائية أو جدولة السوشيال.

| التصنيف | المعلومة |
| --- | --- |
| Confirmed | شريك يتمركز حول المخرجات، التعميد، SLA، والملفات. |
| Assumed | صياغة "Trello + بوابة عميل + اعتماد/SLA" مفيدة داخليا للتقريب لا للتسويق النهائي. |
| Open Question | هل نستخدم مصطلح "Automation Bridge" في التسويق أم يبقى وصفا داخليا؟ |

## 6. Solution

مكونات الحل في MVP:

- إدارة العملاء والعقود والباقات.
- إنشاء مخرجات مرنة بأنواع متعددة.
- ربط كل مخرج بمالك، مساهمين، مواعيد، SLA، وملفات.
- Kanban داخلي للفريق والإدارة.
- تعليقات داخلية وتعليقات عميل مفصولة.
- رفع ملفات ونسخ وربطها بالمخرج.
- تعميد داخلي قبل ظهور المخرج للعميل.
- اعتماد العميل أو طلب تعديل.
- Audit Log لكل قرار مؤثر.
- Dashboard مبسط للعميل وإدارة أوسع للإدارة.

| التصنيف | المعلومة |
| --- | --- |
| Confirmed | هذه المكونات تمثل قلب V1 في AGENTS.md وطلب المالك. |
| Assumed | يمكن إطلاق MVP بدون تكاملات خارجية معقدة. |
| Open Question | هل الإشعارات في V1 داخل المنصة فقط أم بريد إلكتروني أيضا؟ |

## 7. Channels

قنوات الوصول المحتملة:

- اعتماد داخلي داخل سماوة أولا.
- عرض تجريبي لعملاء سماوة الحاليين.
- دراسات حالة بعد تجربة عميلين.
- علاقات مباشرة مع وكالات وخدمات تسويق مشابهة.
- محتوى تعليمي حول ضبط الاعتمادات وSLA في الوكالات.

| التصنيف | المعلومة |
| --- | --- |
| Assumed | القناة الأولى هي التشغيل الداخلي وليس التسويق العام. |
| Open Question | هل توجد نية لإطلاق عام أو Landing Page في V1؟ |

- باقات حسب عدد المخرجات أو حجم التخزين.
- White-label للوكالات.
- إعداد وتشغيل مخصص للفرق الكبيرة.

| التصنيف | المعلومة |
| --- | --- |

| Open Question | هل الإيرادات جزء من هدف V1 أم الهدف الأول تشغيل داخلي وإثبات القيمة؟ |

- استضافة التطبيق وقاعدة البيانات.
- تخزين الملفات والفيديوهات.
- البريد/الإشعارات لاحقا.
- دعم العملاء والتدريب.
- أمن واختبارات وصيانة RLS والصلاحيات.

| --- | --- |

| Open Question | ما حدود حجم الملفات والفيديو في أول تجربة؟ |

## 10. Key Metrics

مقاييس أولية:

- نسبة المخرجات المدخلة داخل شريك من إجمالي مخرجات العميلين التجريبيين.
- نسبة المخرجات ذات مالك وموعد وSLA.
- نسبة الاعتمادات التي تتم من داخل المنصة.
- متوسط وقت انتظار العميل.
- عدد حالات SLA التي توقفت بانتظار العميل دون احتسابها على الفريق.
- عدد المخرجات التي تحتوي Audit Log كامل.
- عدد حالات طلب ملف أو اعتماد خارج المنصة.

| التصنيف | المعلومة |
| --- | --- |
| Confirmed | يجب قياس SLA والاعتمادات وعزل العميل. |
| Assumed | الهدف الأول تقليل التشغيل خارج المنصة وليس الإيراد الفوري. |
| Open Question | ما خط الأساس الحالي: كم مخرج شهريا، كم اعتماد، وكم تعديل؟ |

## 11. Unfair Advantage

مزايا دفاعية محتملة:

- فهم عميق لواقع وكالات التسويق العربية وواجهات RTL.
- تشغيل فعلي داخل سماوة قبل التوسع.
- نموذج مخرج مرن بدل حصر المنتج في منشورات سوشيال.
- فصل صارم بين الداخلي والعميل مدمج من البداية.
- SLA مصمم ليحسب انتظار العميل بشكل عادل.

| التصنيف | المعلومة |
| --- | --- |
| Assumed | التخصص العربي/RTL وواقع سماوة قد يكونان ميزة أولية. |
| Open Question | ما الشيء الذي لا يمكن نسخه بسهولة: البيانات التشغيلية، خبرة سماوة، تجربة عربية، أم شبكة العملاء؟ |

## 12. Riskiest Assumptions

- هل سيستخدم الفريق المنصة يوميا بدلا من واتساب والجداول؟
- هل سيعتمد العميل فعلا من المنصة إذا كانت التجربة بسيطة بما يكفي؟
- هل يمكن ضبط الملفات الكبيرة والتعليقات والنسخ دون تعقيد زائد في V1؟
- هل SLA المبسط يكفي دون Workflow Automation متقدم؟
- هل "مخرج مرن" سيبقى بسيطا أم يتحول لنموذج معقد مبكرا؟
