# UX Vision and Principles: شريك

**المرحلة:** Phase 05 - Information Architecture, UX Model & Role-Based User Flows  
**نوع الوثيقة:** UX Vision & Principles  
**الحالة:** Draft for owner review  
**آخر تحديث:** 2026-06-23  

## 1. الغرض

هذه الوثيقة تحدد رؤية تجربة المستخدم لمنصة شريك. شريك منتج SaaS لإدارة المخرجات والتسليمات، وسماوة هي مالكة المنصة وأول Tenant تشغيلي، وستبدأ التجربة الواقعية مع نحو أربعة عملاء مختلفين.

هذه الوثيقة ليست تصميم واجهات نهائي، وليست Spec، ولا تختار تقنية جديدة.

| التصنيف | القرار |
| --- | --- |
| Confirmed | اسم المنتج هو شريك. |
| Confirmed | سماوة هي المالكة وأول Tenant يستخدم شريك مع عملائه. |
| Confirmed | البداية التشغيلية المتوقعة تقارب 4 عملاء مختلفين. |
| Confirmed | العربية السعودية هي اللغة الأساسية والنبرة سعودية ودودة وعملية. |
| Confirmed | الإنجليزية لاحقا بعد تثبيت العربية، ولا تؤجل RTL أو التوطين السعودي. |
| Confirmed | White Label مؤجل. |

## 2. الشعور المطلوب

يجب أن يعطي شريك شعورا بالوضوح والطمأنينة، لا شعورا بأداة إدارة مشاريع معقدة. العميل يشعر أن سماوة ماسكة الملف، والفريق يشعر أن كل شيء في مكانه، والإدارة تشعر أنها ترى الاختناقات قبل أن تتحول إلى أزمة.

### للعميل

- يعرف وش المطلوب منه الآن.
- يرى النسخة الصحيحة فقط.
- يلقى الملفات والتقارير في مكان واحد.
- يفهم المتبقي من الباقة بدون مصطلحات داخلية.
- يتلقى إشعارا داخل التطبيق عند إضافة تقرير جديد.

### لفريق سماوة

- يعرف وش عليه اليوم.
- يميز بين ملاحظات داخلية وملاحظات عميل.
- يرفع النسخ والملفات بدون خوف من ظهورها للعميل.
- يشوف SLA بصورة عملية، خصوصا ما عاد من العميل للتعديل.

### للإدارة

- ترى العملاء والمخرجات والاختناقات من لوحة واحدة.
- تفرق بين انتظار العميل، انتظار الفريق، وانتظار قرار داخلي.
- تعمد داخليا وترسل للعميل بعد التحقق.
- تراجع سجل النشاط والقرارات عند الحاجة.

## 3. تعريف البساطة والكفاءة والجودة

| المفهوم | العميل | الفريق | الإدارة |
| --- | --- | --- | --- |
| البساطة | أقل خيارات ممكنة، وأوضح زر قرار. | كل سياق المخرج في Drawer واحد. | مؤشرات وأولويات بدل تفاصيل زائدة. |
| الكفاءة | اعتماد أو طلب تعديل من الجوال بسرعة. | رفع، تعليق، وتحويل للمراجعة بأقل تنقل. | فلترة وتدخل سريع حسب العميل وSLA والمالك. |
| الجودة | لا يرى إلا نسخة معتمدة داخليا. | لا تختلط النسخ ولا تضيع الملاحظات. | لا إرسال بلا تعميد ولا قرار بلا Audit. |

## 4. UX Principles

| المبدأ | تعريفه | لماذا مهم | مثال صحيح | مثال خاطئ | التحقق لاحقا |
| --- | --- | --- | --- | --- | --- |
| One Clear Next Action | كل شاشة تعرض الإجراء الأهم للمستخدم حسب حالته. | يقلل أسئلة "وش أسوي الآن؟" | بطاقة عميل تعرض "اعتمد" أو "اطلب تعديل". | عرض خمسة أزرار للعميل في نفس الوقت. | اختبار أن المستخدم يحدد الخطوة التالية خلال 5 ثوان. |
| Role-Relevant Information | كل دور يرى المعلومات اللازمة له فقط. | يحمي البساطة والخصوصية. | العميل يرى حالة مبسطة، والإدارة ترى delay owner. | العميل يرى Kanban داخلي أو Checklist جودة. | مراجعة كل شاشة مقابل الدور والصلاحية. |
| Progressive Disclosure | التفاصيل تظهر عند الحاجة داخل Drawer أو Tab. | يمنع ازدحام الواجهة. | قائمة مخرجات مختصرة، والتفاصيل داخل Drawer. | عرض كل التعليقات والملفات في بطاقة الكارت. | فحص كثافة الشاشات وعدد العناصر الأساسية. |
| Status with Meaning | الحالة تشرح أثرها العملي. | الحالة ليست لون فقط. | "بانتظار موافقتك - وقت SLA متوقف عليك". | "Status 7" أو "Pending" بلا معنى. | اختبار فهم المستخدم للحالة. |
| Visible Ownership | يظهر صاحب المسؤولية أو مالك الخطوة التالية. | يقلل ضياع العمل. | "المسؤول: سارة، الخطوة: مراجعة داخلية". | مخرج نشط بلا owner. | لا مخرج نشط دون مالك ظاهر. |
| Safe Approvals | الاعتماد قرار واضح ومؤكد ومربوط بنسخة. | يمنع اعتماد نسخة خاطئة. | زر اعتماد مع ملخص النسخة والتأكيد. | اعتماد فوري بلا ذكر النسخة. | اختبار أن القرار يسجل actor/version/time. |
| Version Awareness | المستخدم يعرف هل يرى نسخة عمل، نسخة مرسلة، أو نهائية. | يقلل تضارب الملفات. | شارة "النسخة المرسلة للعميل". | ملف "نهائي" داخل التعليقات الداخلية فقط. | مراجعة حالات الملفات والنسخ. |
| Calm Interface | لغة هادئة وودودة بدون تهويل. | تناسب العملاء والفريق تحت الضغط. | "نحتاج موافقتك لإكمال التسليم". | "متأخر جدا بسببك". | مراجعة النبرة العربية السعودية. |
| Mobile-Friendly Client Portal | العميل يستطيع الاعتماد والتعليق من الجوال. | العميل غالبا يتفاعل من الجوال. | أزرار كبيرة ومراجعة ملف سريعة. | جدول عريض لا يعمل على الجوال. | اختبار viewport جوال لكل رحلة عميل. |
| Desktop-Efficient Operations | الفريق والإدارة يحصلون على أدوات كثيفة ومنظمة على سطح المكتب. | التشغيل اليومي يحتاج مسح ومقارنة. | Kanban، جداول، فلاتر، بحث عام. | واجهة ضخمة ببطاقات تسويقية فقط. | اختبار إدارة 4 عملاء و30+ مخرج. |
| No Hidden Internal Leakage | لا يوجد مسار بصري يكشف الداخلي للعميل. | ثقة وأمان. | Internal Only لا يظهر في بوابة العميل. | تحويل تعليق داخلي كما هو إلى تعليق عميل. | اختبار رؤية العميل للتعليقات والملفات. |
| Recoverable Actions | الإجراءات الحساسة لها تأكيد وعودة أو مسار تصحيح. | يقلل الخوف من الخطأ. | رفض انتقال Kanban غير صالح مع rollback. | سحب كارت يرسل للعميل مباشرة. | اختبار invalid transitions. |
| Accessible by Default | كل حالة ومعلومة لها نص وليس لون فقط. | يحقق WCAG 2.2 AA كتخطيط. | شارة "متأخر" مع أيقونة ونص. | تمييز التأخير بالأحمر فقط. | مراجعة contrast، keyboard، labels. |
| RTL-First | العربية وRTL ليست طبقة لاحقة. | التجربة سعودية بالأساس. | ترتيب التنقل والحقول والقراءة من اليمين. | واجهة LTR مترجمة حرفيا. | اختبار كل شاشة بالعربية السعودية. |

## 5. قرارات مالك المنتج المؤثرة على UX

| القرار | التصنيف | أثره على UX |
| --- | --- | --- |
| التقارير تظهر في الملفات، ويضيفها فريق العمل. | Confirmed | لا Tab مستقل للتقارير في بوابة العميل في V1. |
| إشعار داخل التطبيق عند إضافة تقرير جديد. | Confirmed | مركز الإشعارات يعرض "تمت إضافة تقرير جديد". |
| دخول العميل تسجيل عادي، لا Magic Link في V1. | Confirmed | رحلة الدخول تعتمد حساب مستخدم ودعوة. |
| العميل يرى نسخة العميل الحالية وسجل قرارات مبسط، وليس كل تاريخ النسخ الداخلي. | Assumed | يقلل الخطر والتعقيد. |
| إعادة فتح مخرج مسلم تتم بطلب من العميل للفريق/الإدارة، لا ذاتيا بعد التسليم. | Confirmed | زر "طلب متابعة" بدل "إعادة فتح". |
| SLA للعميل مبسط. | Confirmed | تظهر عبارات مفهومة، لا Segments مفصلة. |
| Client Admin أو الإدارة يمكنهم دعوة مستخدمي العميل وفق صلاحيات. | Confirmed | الإعدادات تدعم إدارة/طلب دعوات العميل. |
| Bulk Approval مسموح إذا كانت التجربة جيدة. | Confirmed | يدخل كتحديد متعدد في "بانتظار موافقتي" بشروط. |
| التعليقات المهمة للفريق والعميل تدخل V1، والتعليق الحر هو الأساس. | Confirmed | لا قوائم أسباب إجبارية في V1 إلا عند القرار الحساس. |
| كل شيء مهم يجب أن يعمل من الجوال، خصوصا الإشعارات. | Confirmed | العميل والفريق والإدارة لهم Responsive UX. |
| Global Search مطلوب. | Confirmed | يظهر للفريق والإدارة، وللعميل بحث محدود داخل نطاقه. |
| Bulk Deliverable Creation غير مطلوب في V1. | Confirmed | لا شاشة إنشاء جماعي في UX الأساسي. |
| Activity Feed يحتاج فلاتر مثل الديمو. | Confirmed | فلاتر حسب النوع والحالة والمستخدم والملف. |
| مدير الحساب يمكنه الإرسال للعميل إذا خول وبعد التعميد الداخلي. | Confirmed | تظهر أزرار الإرسال له بشرط الحالة والصلاحية. |
| دعم معاينة كل أنواع الملفات قدر الإمكان. | Confirmed | UX يخطط لمعاينة عامة مع fallback تنزيل. |
| تنزيل عدة ملفات دفعة واحدة غير مطلوب في V1. | Confirmed | لا bulk download في V1. |
| تحويل ملف داخلي إلى Client Visible يكون بإجراء حساس مؤكد. | Assumed | الأفضل UX: Mark Client Visible مع مراجعة الرؤية والتعميد. |

## 6. كيف يقلل شريك الاعتماد على واتساب والبريد والملفات المتناثرة

- كل مخرج له Drawer يحتوي الحالة، النسخة، الملفات، التعليقات، الاعتمادات، وسجل النشاط.
- العميل يدخل مباشرة إلى "بانتظار موافقتي" بدل انتظار رسالة واتساب.
- التقارير ترفع في الملفات مع إشعار داخل التطبيق.
- التعليقات الحرة تحفظ في سياق المخرج أو النسخة.
- Audit يثبت من وافق ومتى وعلى أي نسخة.
- SLA يوضح وقت انتظار العميل دون لغة لوم.
